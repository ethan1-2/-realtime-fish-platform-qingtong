# SAAS 游戏支付抽水平台：Kafka + Flink + Doris 场景题（3 题）

适用背景：
- 你们是 SAAS 平台，服务多个游戏公司（多租户）。
- 平台提供充值/代收/支付回调/补单/退款等能力，通过抽成（抽水）盈利。
- 技术架构：Kafka（事件总线）+ Flink（实时处理）+ Doris（实时数仓/分析）。
- 数据规模：业务事件量极大（可理解为多表日增量达到十亿级以上），存在乱序、迟到、重复投递、热点租户/热点游戏。

目标：
- 用 3 个高频且“能落地”的场景题，把实时数仓最核心的难点一次练透：幂等去重、事件时间/迟到回补、规则版本化、多租户隔离、倾斜治理、可观测与对账可解释。

---

## 场景 1：多租户实时营收与结算（抽成规则可变 + 可对账）

### 业务故事
平台接入 N 个租户（游戏公司），每个租户有多个游戏/区服/渠道。平台代收支付后按规则抽成：
- 抽成可能按租户等级、支付方式、渠道、活动期、阶梯费率等组合计算。
- 支付成功后可能发生退款/拒付/补单，回调可能迟到数小时甚至更久。
- 运营与财务希望：分钟级看大盘；结算时能逐笔对账，解释“为什么这个租户今天应结算金额是 X”。

### 你要交付什么
1. Kafka 事件模型与 topic 规划（最少包含以下事件类型）：
   - pay_success（支付成功）
   - refund_success（退款成功）
   - chargeback（拒付/争议）
   - settlement_adjust（人工调账/补偿）
   - take_rate_rule_change（抽成规则变更，带生效时间与版本）
2. Flink 实时任务：
   - 生成“结算明细”（每笔订单的抽成与应结算金额）
   - 生成“实时指标”（分钟/小时/天：GMV、净入金、平台抽成、租户应结算）
3. Doris 表设计：
   - 明细审计层（用于对账与追溯）
   - 聚合指标层（用于大盘查询与钻取）

### 关键约束与坑点（答题必须覆盖）
- 幂等与去重：
  - Kafka 至少一次投递，Flink 重启会重放，支付回调/补单也可能重复上报。
  - 需要明确“主去重键”是什么（payment_id/order_id/merchant_trade_no 等）以及去重窗口/TTL。
- 事件时间与迟到：
  - 退款/拒付可能跨天迟到；迟到导致历史分钟/小时指标回补。
  - 需要说明 watermark、allowed lateness 的策略，以及“回补”是如何体现在 Doris 中的。
- 抽成规则版本化：
  - 抽成规则会变；必须能解释历史订单按哪版规则计算。
  - 需要讨论：Flink 广播配置 vs 时态维表 join；规则生效时间与事件时间不一致如何处理。
- 多租户隔离与倾斜：
  - tenant_id / game_id / channel_id 分布会极不均匀，头部租户可能压垮某些 key。
  - 需要说明 Kafka 分区键、Flink keyBy 维度、两阶段聚合/加盐等方案。

### 学习抓手（如何高效练）
- 第一步只做两张 Doris 表：
  - `dwd_payment_detail`（明细，幂等写入）
  - `dws_settlement_minute`（分钟聚合）
- 验收标准：
  - 任意挑 100 笔订单，在 Doris 能查到：原始金额、抽成规则版本、抽成金额、应结算金额、退款/拒付更正链路。
  - 人为制造“迟到退款”，分钟指标能被正确更正，且不会翻倍。

## 场景 2：平台级资金风控与异常告警（跨租户共用能力 + 规则可配置）

### 业务故事
你们通过抽成赚钱，最大风险来自资金侧异常：
- 盗刷、羊毛党、异常退款率、某渠道洗量、某租户短时间内交易激增等。
平台希望提供统一的实时风控能力，但不同租户可能有不同阈值、黑白名单与处置策略。

### 你要交付什么
1. Flink 实时风控任务：
   - 输入：支付事件 + 用户登录/注册事件 + 设备指纹/IP + 黑白名单/规则配置变更
   - 输出：`risk_hit` 风险命中事件流到 Kafka（用于下游封禁/二次验证/限额）
2. Doris 风控查询模型：
   - 风险命中明细（可按 tenant_id / risk_type / channel / ip / device_id 检索）
   - 风险分布聚合（按小时/天、租户/渠道等维度看趋势）
3. 规则的可回溯：
   - 每条风险命中都能解释“当时用的规则版本 + 命中证据”

### 关键约束与坑点（答题必须覆盖）
- 多流关联策略：
  - 支付流与登录/设备流如何对齐？interval join、异步维表、还是部分打宽？
  - 数据迟到与缺失时如何降级，避免误报。
- 状态规模控制：
  - 典型特征：10 分钟内同设备注册数、1 小时内退款率、同卡跨租户支付、同 IP 命中数等。
  - 需要说明 state 的 key 设计、窗口、TTL、以及为什么不会爆。
- 倾斜与攻击流量：
  - 黑产流量可能集中在某些 IP 段/渠道，造成 keyBy 热点。
  - 需要说明 anti-skew 手段与准确性保证（分层聚合、加盐、热点隔离、侧输出等）。
- 可观测性：
  - 要定义链路 SLA 指标：Kafka lag、Flink backpressure、checkpoint 时长/失败率、Doris 写入失败率、查询 P95。
  - 告警如何区分“风控规则命中上升”与“数据延迟/重复导致假峰值”。

### 学习抓手（如何高效练）
- 先实现 3 条规则，要求每条都输出 `evidence`（证据字段）让运营看得懂：
  - 频次类：短时间多次支付/失败
  - 金额类：超阈值交易/分段异常
  - 关联类：同设备多账号、同 IP 多租户
- 验收标准：
  - 手工构造一批“重复事件 + 迟到事件”，风控命中数不会因为重放而翻倍。
  - Doris 后台能按单用户/单设备回放最近 24 小时的风险证据链。

## 场景 3：SAAS 平台计费与配额治理（按租户计费 + 保护平台稳定）

### 业务故事
除了抽成，你们可能对租户收 SAAS 服务费：
- 按 API 调用量、成功支付笔数、风控调用次数、报表查询次数、通知推送次数等计费。
同时你们必须做配额与隔离：
- 某租户流量洪峰不能拖垮整个平台；超配额要限流或降级。

### 你要交付什么
1. 租户计费明细与账单：
   - Doris 中可按 tenant_id + dt 查询当天/当月各计费项累计
   - 支持出月账单，并可追溯到明细（争议处理）
2. 实时配额检测与控制流：
   - Flink 实时累计租户用量（QPS、成功笔数、金额等）
   - 超限时输出控制指令（Kafka control topic 或写配置表）供网关/业务系统执行限流
3. 成本可控的分层策略：
   - 哪些数据存明细（审计/争议），哪些只存聚合（趋势/大盘）

### 关键约束与坑点（答题必须覆盖）
- 计费口径与幂等：
  - 计费按“请求”还是按“成功”？遇到重试、补单怎么算？
  - 需要明确去重键与计费事件的生命周期。
- 分区与冷热：
  - Doris 近 7 天高频查询，历史低频；如何分区、TTL、冷热分层，避免写放大与 compaction 压力。
- 资源治理：
  - Kafka 分区规划、Flink 并行度与反压策略、Doris 导入批量/频率与失败重试。
- 数据质量与对账：
  - 要能证明账单不重不漏：抽样对账策略（tenant_id + dt 抽样）与校验指标设计。

### 学习抓手（如何高效练）
- 先做一个目标表：`dws_tenant_daily_bill`（租户日账单聚合）
- 再加一条控制链路：当某租户超过配额，产出 `tenant_throttle` 控制事件
- 验收标准：
  - 人为制造“租户突发流量”，系统能在分钟级发现并触发控制事件，且账单累计不丢不重。

---

## 建议的学习顺序（最省返工）
1. 先做场景 1：幂等去重 + 迟到回补 + 规则版本化（实时数仓硬骨头）。
2. 再做场景 2：在同一条支付明细上叠加风控特征与告警输出（状态与倾斜治理）。
3. 最后做场景 3：补齐多租户治理（计费、配额、隔离、成本控制）。

如果你愿意把练习进一步“工程化”，下一步可以补齐：
- 统一事件字段规范（tenant_id、event_time、trace_id、idempotency_key、rule_version）
- Doris DDL 模板（明细 UNIQUE KEY + 聚合层物化/汇总）
- 端到端 SLA 指标与告警阈值建议
